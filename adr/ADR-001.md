# ADR-001: Single contract vs factory pattern

**Status:** Accepted

**Context:** The original SimplyVest (Solana) uses PDA-derived accounts per stream. On EVM, we could either deploy individual contracts per stream (factory pattern) or use a single contract with mappings.

**Decision:** Single contract with `mapping(bytes32 => Stream)` and `mapping(bytes32 => MilestoneStream)`.

**Rationale:**
- Cheaper deployment (one tx vs one + N)
- Simpler frontend (one address to track)
- All events emitted from one source
- No proxy/upgradeability overhead

**Trade-off:** No per-contract isolation. A bug in the single contract affects all streams.

# ADR-002: Stream ID as keccak256 hash vs sequential counter

**Status:** Accepted

**Context:** Need a unique identifier for each stream.

**Decision:** Use `keccak256(abi.encodePacked(creator, recipient, token, nonce))` for streams and `keccak256(abi.encodePacked("milestone", creator, recipient, token, nonce))` for milestone streams.

**Rationale:**
- Deterministic — frontends can compute ID before tx confirms
- Collision-resistant (keccak256 + nonce)
- Self-documenting (the hash commits to all parameters)

**Trade-off:** Cannot easily enumerate all streams for a recipient without indexing events.

# ADR-003: Status derived from data vs stored enum

**Status:** Accepted

**Context:** Need to track stream lifecycle: Active → Completed/Cancelled.

**Decision:** Store only `cancelled: bool`. Derive `completed` from `amountWithdrawn == amount`.

**Rationale:**
- Single source of truth — no sync bugs between fields
- Fewer state transitions to maintain
- Original Solana design uses same approach

# ADR-004: No upgradeability

**Status:** Accepted

**Context:** Should the contract be upgradeable via proxy?

**Decision:** No proxy. Immutable by design.

**Rationale:**
- Vesting is a trust-sensitive application
- Users need assurance that vesting terms cannot change post-deployment
- If upgrades are needed, deploy a new contract and migrate

# ADR-005: ERC20 transfer at withdraw vs push-pull

**Status:** Accepted

**Context:** Should tokens be pushed to recipient on withdraw, or should recipient pull them?

**Decision:** Push (contract calls `IERC20(token).safeTransfer(recipient, amount)`).

**Rationale:**
- Simpler UX — recipient doesn't need a second tx
- The `withdraw()` is called by the recipient (msg.sender == recipient), so they sign and receive in one step
